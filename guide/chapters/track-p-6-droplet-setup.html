Track P, chapter 6 — Setting up your droplet

This is the chapter where it all comes together: one script, run once, turns your empty droplet into your live website's new home. It asks a series of questions — mostly things you've already saved in your password manager from earlier chapters — and takes care of the rest itself.

Have your password manager open before you start. Nearly every answer this chapter needs, you've already saved: your Cloudflare Tunnel token, team domain, and Access AUD tag from chapter 4, and your Anthropic API key from chapter 5.

  1. Create one more empty, private repository — this one for your automatic nightly backups.

    Where On your organisation's page, click "New," choose "Private," and use the exact name below (if it's taken, a close variant is fine, the same as your organisation's own name in chapter 2). Leave everything else as it is — no README, no template — and click "Create repository." There's nothing to put in it yourself; your droplet fills it in automatically, every night, once it's running.
    ca-state-backup

    You know it worked when… an empty repository named ca-state-backup appears in your organisation's repository list.

  2. Find and note down the "SSH URL" for both your site repository and this new backup repository.

    Where On each repository's own page, click the green "Code" button, then the "SSH" tab (not "HTTPS," which is selected by default), and copy the address shown — it starts with git@github.com:. Do this once for cottage-aesthetics-site and once for ca-state-backup; keep both handy, you'll paste them into the script in a moment.

    You know it worked when… you have two addresses copied, each starting git@github.com: and ending .git.

  3. Open your droplet's console and paste the one setup command.

    Where On your droplet's page in DigitalOcean, click "Console" (top right) — a black, terminal-style window opens right there in your browser tab. Click inside it, then paste the command below and press Enter. Before you copy it, replace <your-org> — it appears twice — with the organisation name you chose in Track P, chapter 2, step 2 (cottage-aesthetics, unless GitHub asked you to use a variant).
    curl -fsSL https://raw.githubusercontent.com/<your-org>/wixy-engine/main/deploy/standalone/setup.sh | WIXY_ENGINE_REPO=https://github.com/<your-org>/wixy-engine.git bash

    You know it worked when… the console starts printing lines of text — it's fetching the setup script and about to start asking you questions.

  4. Answer the script's questions, in the order it asks them.

    You know it worked when… the script stops asking questions and starts printing progress messages of its own — it's moving on to generating your security keys next.

  5. Paste your site repository's new deploy key on GitHub, when the script pauses and asks.

    Where The console prints a block of text starting ssh-ed25519, followed by the exact GitHub page to paste it on. Open that page in a new browser tab, paste the key, tick "Allow write access," and save it. Then go back to the console and press Enter to continue.

    You know it worked when… the new deploy key appears in that repository's list, and the console has moved on to its next question.

  6. Create your AI assistant's own GitHub token, when the script pauses and asks.

    Where The console shows the exact page to visit: github.com/settings/personal-access-tokens/new. Give it any name, set "Repository access" to your site repository only, and set both "Contents" and "Pull requests" permissions to "Read and write." Click "Generate token," copy it, and paste it back into the console.
    Save this in your password manager as "Cottage Aesthetics — AI assistant token".
    This is the credential your AI assistant uses to open pull requests with the changes it suggests — deliberately scoped to nothing more than that. It's exactly why chapter 2's branch protection matters: even this token, in the wrong hands, still can't push straight to your live site.

    You know it worked when… the console accepts your paste and moves on.

  7. Confirm branch protection, twice, when the script pauses and asks.

    Where The console asks you to confirm this is set up on your site repository, then again on your engine fork. You already did both, properly, back in chapter 2 — so for each, just check it's still there if you'd like, then press Enter to move on.

    You know it worked when… the console continues both times without complaint.

  8. Paste your backup repository's new deploy key on GitHub, when the script pauses and asks.

    Where Exactly like step 5, but this time the exact page named is your new ca-state-backup repository's own deploy-keys page. Paste the key shown, tick "Allow write access," save it, then return to the console and press Enter.

    You know it worked when… the key appears on ca-state-backup's deploy-keys list, and the console carries on by itself from here — no more questions.

  9. Watch the script finish, and check its final health report.

    Where The console now works unattended for a minute or two — installing, writing its settings, and starting your website — before running its own built-in check and printing one line per test, each ending [OK] or [FAIL].
    If anything shows [FAIL], it names the exact chapter/step to revisit — fix that one thing and simply paste the whole setup command again; it's safe to re-run from the start as many times as you need, it won't create anything twice.

    You know it worked when… every line ends [OK], and the final line reads something like "5 passed, 0 failed."
