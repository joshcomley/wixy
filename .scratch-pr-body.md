# Mobile edit chrome: immovability, device auto-detect, icon, reveal menu

Operator feedback round (2026-07-21, phone ~365-390px, dark mode) — four reports, one PR.
Decision: `decisions/00084-edit-chrome-immovable-mobile`.

## 1. "The edit area at the bottom can scroll off and then you can't get it back"

The composer and the hours/price control sheets anchored `bottom: 0` to the **layout**
viewport — but the on-screen keyboard and pinch-zoom move the **visual** viewport, so the
chrome could leave the visible region irrecoverably.

- New `editor/src/visualPin.ts`: pins every fixed bottom sheet to the visual viewport
  (`resize`+`scroll` → `bottom`/`left`/width; composer pins internally and releases on
  `destroy()`, sheets pin at their overlay call sites and release on close).
- The overlay appends `interactive-widget=resizes-content` to the preview document's
  viewport meta at startup (preview-only, idempotent, no-op when a site template has no
  meta) so the keyboard resizes the layout viewport itself.
- The admin shell's own viewport meta now locks user scaling
  (`maximum-scale=1, user-scalable=no, interactive-widget=resizes-content`) — the shell
  is an app, not content; outer pinch can no longer pan the slim bar/iframe/composer
  away. Legibility stays served by the in-app zoom + font-scale controls.

## 2. "The default display should be what the user is on — it defaults to desktop always"

Measured live: 390px phone → Mobile ✓, **487px phone → Desktop ✗** (the old
`innerWidth < 480` check misclassifies any phone reporting ≥480 CSS px — display-size
settings, unusual DPRs), 820px tablet → Desktop ✗ (no tablet branch existed).

New `initialDeviceFor(width, coarsePointer)`: <600 → mobile; coarse && <768 → mobile;
coarse && ≤1366 → tablet; fine-pointer <1024 → tablet; else desktop.
Post-fix, measured on the live bundle: **390→Mobile, 487→Mobile, 820→Tablet, 1280→Desktop**.

## 3. "The full screen icon is teeny tiny"

The ⤢/⤡ text glyphs render tiny and faint in Android's system font. Maximize/restore are
now inline SVGs (Feather maximize-2/minimize-2, `currentColor`, 18px) — crisp everywhere.

## 4. "The reveal menu shows below; the gap for it appears above, but it is empty"

Two compounding bugs: `.wx-shell-editing .wx-topbar`'s `visibility: hidden` was never
reset on reveal (the bar slid open as an **empty gap** — layout, no paint), and the nav
lives inside `.wx-body`, **below** the pinned slim edit bar.

- `.wx-shell-chrome-revealed .wx-topbar { visibility: visible }` (later rule wins the
  specificity tie; instant flip so content shows during the slide).
- On ≤720px the nav relocates between the topbar and `.wx-edit-bar-host`
  (`matchMedia`, re-placed live on breakpoint crossings) — desktop's sidebar untouched.
