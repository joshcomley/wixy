# 03 — Project configuration & file layout

This file documents every JSON schema, every well-known file path, every env-var-controlled file, and every directory the Wix CLI reads or writes — both per-project and globally. Skip this and your portal will misidentify project types and corrupt CLI state.

## `wix.config.json` — the linchpin

`wix.config.json` lives at the project root. Its shape determines the flow (see [01-architecture.md](01-architecture.md)). There are **four schema variants**, one per flow. All four use Zod schemas defined in `@wix/cli-*-definitions` and bundled into the CLI's `build/`.

### Variant A — ASTRO flow (current / preferred)

`wix.config.json` for an Astro-based project (whether app or site) follows this discriminated union, keyed on `projectType`:

#### A.1 — ASTRO-SITE (default `projectType`, headless project)

```jsonc
{
  "$schema": "...",                  // optional, IDE hint
  "projectType": "Site",             // default if omitted
  "appId":  "00000000-0000-0000-0000-000000000000",
  "siteId": "00000000-0000-0000-0000-000000000000"
}
```

Both `appId` and `siteId` are required. `projectType` defaults to `"Site"` when omitted.

#### A.2 — ASTRO-APP

```jsonc
{
  "$schema": "...",
  "projectType": "App",              // discriminator literal — required for App branch
  "appId":  "00000000-0000-0000-0000-000000000000",
  "projectId":      "...",           // optional
  "codeIdentifier": "...",           // optional, JS-compatible identifier
  "namespace":      "@prefix/suffix" // optional, set once at scaffold
}
```

The CLI also needs an `astro.config.{js,cjs,mjs,ts}` next to `wix.config.json` for either Astro variant. Without that file, the CLI falls through to the legacy APP flow.

### Variant B — Legacy APP flow

```jsonc
{
  "appId": "00000000-0000-0000-0000-000000000000"
}
```

Only `appId` is required. The legacy `@wix/cli-app` package, installed in the project's own `node_modules`, defines its own schema and reads additional fields lazily.

### Variant C — Modern SITE flow

```jsonc
{
  "siteId": "00000000-0000-0000-0000-000000000000"  // UUID
}
```

`siteId` only. Validated as a UUID.

### Variant D — Legacy SITE / Velo flow

```jsonc
{
  "siteId":    "00000000-0000-0000-0000-000000000000",
  "uiVersion": "..."                  // any string; presence selects the legacy flow
}
```

`uiVersion` is the marker that flips to the legacy `cli-site-old` code path.

### Variant E — SITE-LIVE (env-var gated)

```jsonc
{
  "siteId":    "00000000-0000-0000-0000-000000000000",
  "veloAppId": "00000000-0000-0000-0000-000000000000"
}
```

Only loaded when `WIX_CLI_SITE_LIVE=true` is in the environment. Exposes ONE command (`wix dev`).

### UNKNOWN flow

If `wix.config.json` is missing OR malformed OR fails the schema for its flow:

- `wix --help` still lists the global commands.
- Any non-global command produces `FailedToIdentifyProgramFlow` with a hint about which schema field is missing.

For Wixy's project-type detector: read `wix.config.json` raw with `fs.readFile` + `JSON.parse`, then apply the cascade in [01-architecture.md § decision table](01-architecture.md#decision-table-live-from-source). Don't shell out to `wix --help` to detect — it's slow and racy.

## `.env.local` — environment variables (ASTRO flow)

ASTRO projects read `.env.local` at the project root via `dotenv`-style parsing. Wix's bundled schema only declares one well-known key:

```env
WIX_SITE_ID=00000000-0000-0000-0000-000000000000   # optional, UUID
```

Other keys are passed through. `wix env pull` populates this file by fetching server-side env vars and merging (not replacing).

`.env.local` is intended to be git-ignored. There's no `.env` (without `.local`) convention enforced.

## `wix.lock` — dependency lockfile

`wix.lock` (at the project root) is the CLI's own lockfile, tracking installed Wix packages and their resolved versions. Generated by `wix install` and `wix uninstall` (SITE flows) and by the scaffolder. **Don't edit by hand.** Wix CLI silently mutates it; mismatches between `wix.lock` and `package.json` trigger reinstalls.

## `.wix/` — per-project state (gitignored)

A `.wix/` directory at the project root holds CLI-private state. The CLI creates it on demand. **Always git-ignore it.**

```
<project>/.wix/
  debug.log                              # ASTRO + global crash report destination
  debug-logs/cli-debug.log               # ASTRO commands' verbose log
  build-metadata.json                    # ASTRO build output meta (paths, extension topology)
  topology.json                          # ASTRO deployment topology (preview/production URLs)
  app.config.json                        # ASTRO app config (siteId, devArmTag)
  multilingual/
    metadata.json                        # translation push/pull metadata
    translations/                        # downloaded translations cache
```

Per-project `.env.local` lives **at the root**, not inside `.wix/`.

### `build-metadata.json` (write side, by `wix build`)

```ts
{
  appManifestPath: string;
  outDir: string;
  clientDir: string;
  serverDir?: string;
  staticsUrlPlaceholder?: string;
  serverUrlPlaceholder?: string;
  generateAppDeploymentGroupId?: boolean;
  extensionsTopology?: ExtensionTopologyItem[];
}
```

Used by `preview`/`release` to know what to upload.

### `topology.json` (read side, populated by Wix server)

```ts
Record<"production" | "preview" | "development", {
  id: string;                           // ARM tag identifier
  site?:      { main: string };         // site URL
  dashboard?: { main: string };         // dashboard URL
  editor?:    { main: string };         // editor URL
}>
```

For Wixy: read this to know what URLs to surface to the user after a release/preview.

## Source layout (ASTRO)

```
<project>/
  wix.config.json
  package.json
  tsconfig.json
  astro.config.mjs              # signals ASTRO flow
  .nvmrc                        # Node version pin
  env.d.ts                      # auto-generated TS types for env vars
  src/
    extensions/                 # one folder per generated extension
      <id>/
        extension.ts            # registration (uses `extensions.dashboardPage({...})` etc.)
        page.tsx / widget.tsx   # implementation, depends on type
    translations.json           # i18n source for wix translation push
  public/                       # static assets served at root
  dist/                         # build output (gitignored)
  .wix/                         # CLI state (gitignored)
  .env.local                    # env (gitignored)
  node_modules/
```

For the legacy APP layout the structure is similar but the project leans on `@wix/cli-app`'s own conventions: extensions under `src/extensions/{dashboard,site,backend}/`, and a different `extension.ts` flavour.

## Global config: `~/.wix/user.config.json`

Schema (Zod):

```ts
{
  telemetry: boolean,    // default true — toggled by `wix telemetry on|off`
  tunneling: boolean     // default false — toggled implicitly by sessions using --tunnel
}
```

Lives at `<homedir>/.wix/user.config.json`. If absent, defaults apply. Editable safely as JSON.

## Global cache: `~/.wix/version.cache.json`

The CLI caches the latest-published `@wix/cli` version here and uses it to print an `OutdatedVersionMessage` at the end of any command if you're behind. Refresh interval is internal to `versionNotifier`.

## Global crash log: `~/.wix/debug.log`

Most recent stack trace from a failed command. Wix asks you to attach this when filing bugs. Per-project `.wix/debug.log` overrides for project-specific crashes.

## What the CLI **never** stores

- Source code or build artefacts globally — everything is per-project under `.wix/` or `dist/`.
- Plaintext API secrets in `wix.config.json` — secrets are server-side and pulled with `wix env pull` into `.env.local`.
- Authentication for a Wix account in `wix.config.json` — auth lives globally under `~/.wix/auth/`.

For Wixy, that means: **`wix.config.json` is safe to read, parse, and even commit. `~/.wix/auth/` is NEVER safe to share.**
